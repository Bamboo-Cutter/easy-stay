# Easy-Stay Hotel 小组演示文档（约 10 分钟）

适用项目：`easy-stay-hotel`（后端 API + 用户端 H5 + 商户/管理员后台）

## 1. 演示目标（开场 20 秒）

在 10 分钟内向老师/评审清楚展示：
- 我们做的是什么（酒店预订系统）
- 系统有哪些角色（用户 / 商户 / 管理员）
- 核心业务链路是否跑通（上架 -> 审核 -> 搜索下单）
- 技术实现亮点（NestJS + Prisma + PostgreSQL，多端联动）

## 2. 演示角色分工（两个人，按功能分工）

建议按“平台管理侧”与“用户业务侧”拆分，逻辑最清晰：

- A 同学（平台侧）：项目背景 + 架构总览 + 商户端 + 管理员端（约 5 分钟）
- B 同学（用户侧）：用户端完整预订链路 + 总结（约 5 分钟）

这样分工的好处：
- A 负责“供给侧”流程：酒店内容生产、审核、库存治理
- B 负责“消费侧”流程：搜索、选房、下单、订单结果
- 前后衔接自然，能完整体现业务闭环

## 3. 演示前准备（提前 10-15 分钟完成）

### 3.1 启动服务

在项目根目录执行（按需）：

```bash
docker compose up -d
pnpm -C apps/api start:dev
pnpm -C admin-web dev
pnpm -C user-mobile dev:h5
```

默认地址：
- 后端 API：`http://localhost:3000`
- 管理后台：`http://localhost:5173`
- 用户端 H5：`http://localhost:10086`

### 3.2 演示账号（seed 默认）

- 管理员：`admin@demo.com / Admin12345!`
- 商户：`merchant1@demo.com / Merchant12345!`
- 用户：`user@demo.com / User12345!`

### 3.3 彩排检查清单（重要）

- `GET /health` 可返回 `ok`
- 商户端能登录并看到酒店列表
- 管理员端能看到待审核/酒店管理列表
- 用户端首页/酒店列表能加载数据
- 至少准备 1 个可下单酒店（避免现场临时造数据）

## 4. 10 分钟演示流程（推荐脚本，双人版）

## 0:00 - 1:30 项目介绍（A）

讲解要点：
- 这是一个“酒店预订全流程”演示系统，不是单一前端页面
- 包含三端：
  - `apps/api`：NestJS + Prisma + PostgreSQL
  - `admin-web`：商户/管理员后台（React + Vite）
  - `user-mobile`：用户端（Taro React，演示 H5）
- 目标是打通真实业务链路：商户上架、管理员审核、用户搜索并预订

可配合展示：
- 根目录结构（IDE 或文件树）
- `README.md` 中技术栈/模块说明

## 1:30 - 3:45 商户端演示（A）

建议讲“商户如何把酒店投放到平台”。

演示步骤（选择最稳的一条）：
- 登录后台（商户账号）
- 进入商户酒店列表（`MerchantHotelList`）
- 打开一个已有酒店进行编辑（比现场新建更稳）
- 展示酒店基础信息、图片/标签、房型信息
- 展示价格日历维护（`CalendarEdit`）
- 将酒店状态改为“提交审核 / 待审核”（如界面支持）

讲解要点：
- 商户负责内容维护与价格配置
- 后端按房型 + 日期维护价格和库存（便于后续查询/预订）
- 价格日历适合酒店场景，支持不同日期差异化定价

备用方案（防止新增流程太慢）：
- 直接展示已存在酒店的“编辑 + 价格维护 + 提交审核”

## 3:45 - 5:00 管理员端演示（A）

建议讲“平台如何做审核和库存管理”。

演示步骤：
- 退出并登录管理员账号
- 进入管理员酒店列表 / 待审核列表（`AdminHotelList`）
- 打开某个待审核酒店
- 执行审核通过（或拒绝并说明理由）
- 展示库存管理（按房型日期查看/设置库存，如界面已有入口）

讲解要点：
- 管理员和商户角色权限分离
- 酒店上架前有审核流程，避免脏数据直接展示给用户
- 库存按日管理，可支持预订时实时扣减/取消释放

## 5:00 - 9:00 用户端演示（B）

建议讲“最终用户如何完成一次预订”。

演示步骤（主链路）：
- 打开用户端 H5 首页 / 酒店列表
- 搜索城市或筛选酒店（如“上海”）
- 进入酒店详情页
- 查看房型报价/可订情况（`roomOffers`）
- 选择入住/离店日期，进入预订页
- 提交订单（`POST /bookings` 对应链路）
- 到个人中心/订单页查看结果（`profile` / `booking` 页面）

讲解要点：
- 前端不是写死数据，连接真实后端接口
- 报价与可用库存来自后端日历/库存表计算
- 预订成功后会扣减库存，取消订单会释放库存（后端已实现）

备用方案（若现场网络/端口异常）：
- 先演示管理后台，再用 Postman/curl 展示 `GET /hotels`、`GET /hotels/:id/offers`、`POST /bookings`

## 9:00 - 10:00 总结（B，A 补充）

建议总结模板（可直接说）：

“我们的项目重点不是单页展示，而是完整的多角色业务闭环。前半部分我们展示了平台侧能力：商户维护酒店与价格、管理员审核与库存管理；后半部分展示了用户侧能力：搜索、选房、预订。技术上我们使用 NestJS + Prisma + PostgreSQL 做后端，React/Taro 做前端，围绕酒店按日价格和库存模型实现核心业务。” 

## 5. 现场台词参考（精简版）

### A（开场）

“大家好，我们组做的是一个酒店预订演示系统。和普通前端展示不同，我们做了三端联动：用户端、商户/管理员后台，以及后端 API。我们两个人按功能分工演示：我先讲平台侧流程（商户和管理员），另一位同学再讲用户侧预订流程，最后做总结。” 

### A（商户端 + 管理员端）

“现在我用商户账号登录后台。商户可以维护酒店基础信息、房型、图片标签，并按日期配置价格。这里的价格日历是酒店业务核心，因为节假日和平日价格会不同。配置完成后，商户提交审核，等待管理员处理。” 

“接下来切换管理员角色。管理员可以查看待审核酒店并进行通过/拒绝，同时也可以按房型和日期管理库存。这样平台能够控制上线质量，并保证后续预订时库存计算准确。下面交给另一位同学演示用户侧下单流程。” 

### B（用户端 + 总结）

“最后来到用户端。用户可以浏览酒店、查看详情和房型报价，选择入住离店日期后提交订单。这个报价和可订状态来自后端实时查询，不是前端写死。下单后订单会进入记录，同时后端会扣减对应日期库存。” 

“总结一下，我们两个人分别展示了平台侧和用户侧功能，合起来就是完整的酒店预订业务闭环。” 

## 6. 常见问题与回答（答辩备用）

- 问：你们的亮点是什么？
  - 答：不是单端页面，而是三角色业务闭环；并且用了酒店场景常见的“按日期价格/库存”模型。

- 问：为什么拆成三端？
  - 答：角色职责不同，用户端关注下单体验，商户端关注内容与价格维护，管理员端关注审核与平台治理。

- 问：数据库如何设计库存？
  - 答：按 `room_id + date` 维护每日库存（`room_inventory_daily`），预订时按日期区间校验并扣减。

- 问：还能扩展什么？
  - 答：支付回调、退款、评价明细、优惠券、订单状态流转、消息通知等。

## 7. 演示失败兜底（务必准备）

- 若 `user-mobile` 页面打开失败：改用后台 + API 演示主链路
- 若管理员/商户登录异常：用 seed 账号重新登录，必要时切换备用浏览器
- 若个别操作接口报错：展示 API 文档 `apps/api/API.md` 和已实现接口列表，说明是演示环境数据状态问题
- 若时间不足：保留“商户提交审核 -> 管理员通过 -> 用户下单”三步，跳过细节编辑

## 8. 演示当天建议（很实用）

- 一台电脑开三窗口：后台、用户端、API/终端
- 组员发言按顺序接力，避免抢话
- 每段只讲“目标 + 操作 + 结果”，不要陷入代码细节
- 代码实现细节放到老师追问时再展开（如 Prisma 模型、库存扣减逻辑、权限控制）
